# Background

Many specifications share common structure: standard build steps (linting, testing, documentation), recurring task groups, and boilerplate front matter. Currently, every new spec starts from scratch, requiring manual setup of these repeated patterns.

Additionally, task group identifiers are limited to plain letters (A, B, C, ...). Users have expressed interest in using emoji characters as group identifiers (e.g., ðŸ§ª for tests) for more expressive and scannable implementation plans, along with automatic subtask numbering to eliminate manual bookkeeping.

# Proposal

## Template System

Introduce a Markdown-based template system for `tinyspec new`. Templates are regular spec Markdown files that serve as starting points for new specs.

**Storage (two-tier, repo takes precedence):**

- Repo-level: `.specs/templates/`
- User-level: `~/.config/tinyspec/templates/`
- On name conflicts, the repo-level template wins.

**Selection:**

- If a `default` template exists (e.g., `default.md`), it is auto-applied by `tinyspec new` unless overridden.
- The `--template <name>` flag selects a specific template explicitly.
- The default template is optional â€” if none exists, `tinyspec new` behaves as it does today.

**Substitution:**

- Dynamic values like title and date are substituted into the template content before writing.

## Emoji Task Group IDs
