# Background

Tinyspec is useful as a standalone tool, but it lives in isolation from the rest of a team's workflow. When a spec completes, nothing else knows about it. When a task is checked, there's no signal to external systems. This limits tinyspec's usefulness in team environments where work is tracked in Linear, Jira, Slack, GitHub Issues, or custom dashboards.

The problem is not that tinyspec should integrate with each of these systems directly — that's scope creep. The problem is that tinyspec has no event surface at all. There's nowhere to plug in.

# Proposal

Add a hook system to `~/.tinyspec/config.yaml` (and optionally a project-level `.tinyspec.yaml`) that maps lifecycle events to shell commands. When an event fires, tinyspec executes the configured commands with environment variables describing the event.

**Events:**

|Event|When it fires|
|-----|-------------|
|`on_task_check`|A task is marked complete via `tinyspec check`|
|`on_task_uncheck`|A task is unmarked via `tinyspec uncheck`|
|`on_spec_complete`|All tasks in a spec are checked (status → Completed)|
|`on_spec_start`|First task in a spec is checked (status → InProgress)|
|`on_spec_create`|A new spec file is created via `tinyspec new`|

**Environment variables passed to hook commands:**

|Variable|Description|
|--------|-----------|
|`TINYSPEC_EVENT`|Event name (e.g., `on_spec_complete`)|
|`TINYSPEC_SPEC`|Spec name (kebab-case)|
|`TINYSPEC_SPEC_TITLE`|Spec title from front matter|
|`TINYSPEC_SPEC_GROUP`|Group name, or empty string|
|`TINYSPEC_TASK_ID`|Task ID (for task events only, e.g., `A.1`)|
|`TINYSPEC_SPEC_PATH`|Absolute path to the spec file|

**Config format:**

```yaml
hooks:
  on_spec_complete:
    - 'echo "Spec $TINYSPEC_SPEC_TITLE completed" | slack-notify "#dev"'
    - 'linear-cli complete --title "$TINYSPEC_SPEC_TITLE"'
  on_task_check:
    - './scripts/update-dashboard.sh'
```

Hooks are executed sequentially in the order listed. A hook failure (non-zero exit) prints a warning but does not block the tinyspec operation that triggered it — hooks are side effects, not gatekeepers.

Project-level `.tinyspec.yaml` hooks are merged with user-level config, with project hooks running first. This allows repo-specific hooks (e.g., updating a project board) to coexist with user-level hooks (e.g., personal notifications).

A `--no-hooks` flag on `check`, `uncheck`, and `new` suppresses hook execution for that invocation.

# Implementation Plan

- [x] A: Extend config schema to support hooks
  
  - [x] A.1: Add `hooks: HashMap<String, Vec<String>>` to the `Config` struct
  - [x] A.2: Define an `Event` enum with all supported event variants
  - [x] A.3: Parse hooks from `~/.tinyspec/config.yaml` and optionally from `.tinyspec.yaml` at the project root
  - [x] A.4: Implement config merging: project-level hooks run before user-level hooks for the same event
- [x] B: Implement hook execution engine
  
  - [x] B.1: Write a `run_hooks(event: Event, context: HookContext)` function that looks up configured hooks and runs them
  - [x] B.2: Build `HookContext` from the current spec/task state and set it as environment variables before each hook command
  - [x] B.3: Execute hook commands via `sh -c`, capture stderr/stdout, print warnings on non-zero exit
  - [x] B.4: Add `--no-hooks` flag to `check`, `uncheck`, and `new` subcommands
- [x] C: Fire hooks from the appropriate commands
  
  - [x] C.1: Fire `on_task_check` and `on_spec_start` / `on_spec_complete` from `tinyspec check`
  - [x] C.2: Fire `on_task_uncheck` from `tinyspec uncheck`
  - [x] C.3: Fire `on_spec_create` from `tinyspec new`
  - [x] C.4: Detect status transitions (Pending → InProgress, InProgress → Completed) to fire the correct spec-level events
- [x] D: Add `tinyspec hooks test` command
  
  - [x] D.1: Add `hooks test <event>` subcommand that fires a named event with dummy context data
  - [x] D.2: Print each hook command and its exit code/output to help users debug hook configuration
- [x] E: Documentation
  
  - [x] E.1: Add a `# Hooks` section to `CLAUDE.md` with the full event list, env var reference, and config examples
  - [x] E.2: Add hooks config example to the output of `tinyspec init`
